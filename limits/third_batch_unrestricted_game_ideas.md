# 第三批：十个完全不同的小游戏方案（不限语言，可用 C++ / C# / TypeScript / Go / Lua / Godot 等）

> 目标：这一批不再限制 Java / Python / Rust / R，语言和技术栈完全自由选择。  
> 这 10 个方案刻意避开前两批的题材：不做卡牌、迷宫、打字、奶茶店经营、生存避障、怪物成长、节奏点击、数独、投篮、交通灯调度、侦探证据板、炼金、星图航线、基因花园、拍卖、密码密室、机器人流水线、钓鱼生态、图书馆分类、调色挑战。  
> 每个方案都必须有可视化前端或可操作入口，不能只是命令行 demo。

---

## 总体实现建议

### 可选技术栈方向

| 类型 | 推荐技术 |
|---|---|
| 桌面 2D 游戏 | C++ + SDL2 / SFML / Raylib |
| 桌面工具型游戏 | C++ + Dear ImGui / Qt |
| Web 游戏 | TypeScript + Phaser / PixiJS / Three.js |
| 物理游戏 | C++ + Box2D / Unity C# / Godot |
| 移动端游戏 | Unity C# / Flutter + Flame / Swift + SpriteKit |
| 在线多人原型 | TypeScript 前端 + Go WebSocket 服务端 |
| 可扩展脚本 | Lua / Wren / JavaScript 脚本层 |
| 高性能模块 | C++ / Rust / Zig / Go |

### 给 Codex 的通用目标

每个游戏都建议先做 MVP：

1. 先实现一个可以看、可以操作的前端。
2. 再实现最小游戏规则。
3. 再加入关卡、分数、存档、编辑器、排行榜或调参面板。
4. 多语言项目优先使用文件 JSON、HTTP、WebSocket 或动态库边界，避免一开始就做复杂绑定。

# 1. 桥梁应力建造游戏《Bridge Stress Lab》

## 游戏概念

玩家在峡谷两端之间搭桥，让小车从左侧驶到右侧。玩家可以放置木梁、钢梁、绳索和支点。点击测试后，系统会模拟桥梁受力，桥可能弯曲、断裂或坍塌。

这不是单纯搭积木，而是一个“结构设计 + 物理测试”的小游戏。玩家需要在预算有限的情况下，让桥足够稳但不要过度浪费材料。

## 推荐语言组合

```text
C++ + Box2D + Dear ImGui
```

## 可视化前端 / 可操作入口

使用 **C++ + Dear ImGui + SDL2** 做桌面窗口：

- 左侧工具栏：选择木梁、钢梁、绳索、删除工具。
- 中间画布：鼠标点击两个节点创建梁。
- 右侧属性面板：显示预算、材料强度、当前桥梁重量。
- 底部按钮：`Build Mode`、`Test Mode`、`Reset`。
- 测试时显示小车移动和梁的变形。
- 梁接近断裂时高亮显示。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 窗口、工具栏、编辑器 | C++ / Dear ImGui |
| 物理模拟 | Box2D |
| 关卡数据 | JSON |
| 构建系统 | CMake |

## 玩法规则

- 每种材料有价格、最大拉力、最大压力。
- 玩家总预算有限。
- 小车重量逐关增加。
- 桥梁断裂则失败。
- 成功后根据预算剩余和稳定程度评分。

## MVP 功能

- 1 个固定峡谷关卡。
- 支持创建和删除梁。
- 支持测试小车通过。
- 桥梁超过受力阈值会断裂。
- 成功或失败弹窗。

## 数据文件示例

```json
{
  "level_id": "bridge_001",
  "budget": 500,
  "anchors": [[0, 300], [800, 300]],
  "vehicle_weight": 120,
  "materials": {
    "wood": {"cost": 10, "strength": 80},
    "steel": {"cost": 25, "strength": 180}
  }
}
```

## Codex 开发任务建议

```text
1. 创建 CMake C++ 项目，引入 SDL2、Dear ImGui、Box2D。
2. 实现 BridgeEditor，支持鼠标点击添加节点和梁。
3. 实现 Material 数据结构，包含 cost、strength、density。
4. 将玩家搭建结果转换为 Box2D joint/body 模拟对象。
5. 点击 Test 后生成小车刚体并开始模拟。
6. 当梁的受力超过阈值时删除该梁并播放断裂效果。
7. 保存和读取 level_001.json。
```

# 2. 潜艇声呐躲猫猫《Submarine Sonar Hunt》

## 游戏概念

玩家驾驶潜艇在黑暗海域中行动。屏幕上不是完整地图，而是声呐扫描出的模糊回波。玩家需要根据声波回传判断障碍物、敌舰、宝箱和出口位置。

游戏重点是“有限信息导航”。玩家看不清环境，只能靠周期性声呐、声音提示和回波图形判断风险。

## 推荐语言组合

```text
Godot / GDScript + C++ GDNative 模块
```

## 可视化前端 / 可操作入口

使用 **Godot 2D**：

- WASD 控制潜艇移动。
- 鼠标或空格键发射声呐脉冲。
- 屏幕中心显示潜艇，周围只有声呐扫过后的短暂轮廓。
- 右侧仪表显示氧气、电量、噪音等级。
- 敌舰靠近时有方向性低频音效。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 2D 场景、UI、输入、音效 | Godot / GDScript |
| 声呐回波计算 | C++ GDNative 或 GDExtension |
| 关卡编辑 | Godot TileMap |
| 数据格式 | `.tres` / JSON |

## 玩法规则

- 声呐会暴露玩家位置，敌舰会追踪声源。
- 静默航行更安全，但看不清路。
- 撞到礁石会损失耐久。
- 找到出口或打捞足够物资后通关。

## MVP 功能

- 一个黑暗海底地图。
- 玩家能移动和发射声呐。
- 声呐能显示附近障碍物轮廓。
- 一个敌舰会向最近一次声呐源移动。
- 到达出口通关。

## 数据文件示例

```json
{
  "map_id": "deep_sea_001",
  "player_start": [4, 12],
  "exit": [30, 3],
  "reefs": [[10, 10], [11, 10], [12, 10]],
  "enemy_subs": [[20, 8]],
  "oxygen_seconds": 180
}
```

## Codex 开发任务建议

```text
1. 在 Godot 中创建 2D 场景，加入 PlayerSubmarine 节点。
2. 实现 WASD 移动、氧气倒计时和耐久显示。
3. 实现 sonar_ping()，在半径内短暂显示障碍物轮廓。
4. 用 C++ 实现快速 raycast / echo sampling 模块。
5. 实现 EnemySub，根据最近一次声呐位置移动。
6. 加入出口区域和通关 UI。
```

# 3. 影子剧场解谜《Shadow Puppet Theater》

## 游戏概念

玩家在舞台上摆放木偶、齿轮、灯光和遮挡物，让墙上的影子刚好形成目标图案。每一关给出一个剪影目标，玩家需要调整光源角度、物体位置和旋转角度。

它的核心不是颜色或绘画，而是“光照投影 + 空间推理”。

## 推荐语言组合

```text
TypeScript + Three.js + Rust WebAssembly
```

## 可视化前端 / 可操作入口

使用 **浏览器 3D / 2.5D 画布**：

- 左侧零件库：圆形、三角形、人物木偶、齿轮、遮挡板。
- 中间舞台：拖动物体、旋转物体、调整高度。
- 右侧墙面：实时显示投影剪影。
- 顶部目标：显示需要匹配的影子轮廓。
- 点击 `Check` 计算相似度并给出星级。

## 各模块分工

| 模块 | 技术 |
|---|---|
| Web 前端、拖拽、UI | TypeScript |
| 3D / 光照 / 投影展示 | Three.js |
| 轮廓相似度计算 | Rust 编译为 WASM |
| 关卡数据 | JSON |

## 玩法规则

- 每关有目标剪影。
- 玩家可使用的物体数量有限。
- 投影相似度超过 85% 即可通关。
- 更少零件、更高相似度可获得更高评分。

## MVP 功能

- 3 种基础形状。
- 1 个可拖动光源。
- 实时显示影子。
- WASM 计算当前影子和目标影子的相似度。
- 5 个关卡。

## 数据文件示例

```json
{
  "level_id": "shadow_001",
  "target_mask": "assets/targets/rabbit.png",
  "allowed_shapes": ["circle", "triangle", "rectangle"],
  "max_objects": 5,
  "pass_score": 0.85
}
```

## Codex 开发任务建议

```text
1. 创建 Vite + TypeScript 项目，引入 Three.js。
2. 实现舞台、墙面、光源和基础几何体。
3. 支持鼠标拖动和旋转物体。
4. 将渲染出的影子导出为低分辨率 mask。
5. 用 Rust 实现 compare_masks(a, b) -> similarity，编译为 WASM。
6. TypeScript 调用 WASM，显示相似度和星级。
```

# 4. 时间回放平台跳跃《Time Loop Platformer》

## 游戏概念

这是一个平台跳跃游戏。玩家需要到达终点，但某些机关必须由“过去的自己”协助完成。每次玩家按下回放按钮，上一轮的行动会变成一个幽灵角色自动重演。

玩家要设计自己的行动序列，让多个时间分身配合踩按钮、挡激光、开门。

## 推荐语言组合

```text
Unity C# + Lua 脚本关卡逻辑
```

## 可视化前端 / 可操作入口

使用 **Unity 2D**：

- A/D 移动，空格跳跃。
- R 键开始下一轮时间循环。
- 幽灵角色以半透明方式重放上一轮操作。
- 场景中有按钮、门、移动平台、激光。
- 右上角显示当前循环编号和剩余时间。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 2D 平台动作、物理、动画 | Unity C# |
| 关卡机关脚本 | Lua |
| 输入录制和重放 | C# |
| 关卡配置 | JSON / ScriptableObject |

## 玩法规则

- 每轮时间限制 20 秒。
- 玩家行动会被记录为输入序列。
- 下一轮会生成一个幽灵重放上一轮输入。
- 玩家最多拥有 3 个幽灵。
- 所有按钮被同时压住时出口开启。

## MVP 功能

- 一个小型平台关卡。
- 输入录制与回放。
- 至少 1 个按钮和 1 扇门。
- 一个幽灵可以重复上一轮动作。
- 到达出口通关。

## 数据文件示例

```json
{
  "level_id": "loop_001",
  "duration_seconds": 20,
  "max_ghosts": 3,
  "objects": [
    {"type": "button", "id": "b1", "position": [4, 1]},
    {"type": "door", "id": "d1", "requires": ["b1"], "position": [12, 1]}
  ]
}
```

## Codex 开发任务建议

```text
1. 创建 Unity 2D 项目，实现 PlayerController。
2. 实现 InputRecorder，按固定 tick 记录左右移动、跳跃、交互。
3. 实现 GhostPlayer，根据记录输入重放动作。
4. 实现 Button 和 Door 机关。
5. 接入 Lua 或先用 JSON 描述机关规则。
6. 做一个 loop_001 测试关卡。
```

# 5. 弹珠魔法台《Pinball Spell Forge》

## 游戏概念

玩家操作弹珠台左右挡板，让魔法弹珠击中不同符文。符文组合会释放法术，例如火球、冰冻、连锁闪电。目标是在限定球数内激活指定法术组合。

它不是节奏游戏，也不是投篮游戏，而是基于弹珠物理和符文组合的动作解谜。

## 推荐语言组合

```text
C++ + SFML + Lua
```

## 可视化前端 / 可操作入口

使用 **SFML** 做桌面 2D：

- 左右 Shift 控制挡板。
- 空格发射弹珠。
- 弹珠台上有符文、弹簧、传送门、倍率区。
- 右侧显示已激活符文序列。
- 符文连成指定组合时播放法术动画。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 渲染、输入、音效 | C++ / SFML |
| 简化弹珠物理 | C++ |
| 符文组合和关卡脚本 | Lua |
| 资源配置 | TOML / JSON |

## 玩法规则

- 弹珠击中符文会记录符文序列。
- 不同序列触发不同法术。
- 法术可以改变场地，例如冻结挡板、复制弹珠、打开通道。
- 每关有目标法术。

## MVP 功能

- 一个弹珠台场景。
- 左右挡板可操作。
- 弹珠能碰撞反弹。
- 5 个符文点位。
- Lua 定义 3 种符文组合。

## 数据文件示例

```lua
spells = {
  {combo = {"fire", "air", "fire"}, result = "fireball", score = 300},
  {combo = {"ice", "water"}, result = "freeze_gate", score = 200},
  {combo = {"spark", "spark", "air"}, result = "chain_lightning", score = 500}
}
```

## Codex 开发任务建议

```text
1. 创建 C++ SFML 项目，绘制弹珠台、弹珠和挡板。
2. 实现基础圆形碰撞和反弹。
3. 实现左右挡板旋转控制。
4. 加入 RuneTarget，弹珠碰到后追加符文。
5. 集成 Lua，读取 spells.lua。
6. 当符文序列匹配 combo 时触发动画和得分。
```

# 6. 灾区无人机救援《Drone Swarm Rescue》

## 游戏概念

玩家作为救援调度员，控制多架小型无人机在灾区地图中寻找幸存者、投递物资、避开浓烟和断电区域。每架无人机有电量、载重和传感器范围。

它和交通灯调度不同，核心是多单位任务分配、视野探索和资源救援。

## 推荐语言组合

```text
TypeScript + PixiJS + Go WebSocket 服务端
```

## 可视化前端 / 可操作入口

使用 **浏览器 Canvas**：

- 玩家在地图上点击无人机，再点击目标点设置任务。
- 地图有雾区，只有无人机扫过的区域可见。
- 右侧面板显示每架无人机电量、任务、载重。
- 服务器每秒推送地图状态。
- 支持单人本地，也可以扩展成多人协作。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 地图渲染、拖拽、任务 UI | TypeScript / PixiJS |
| 状态同步、多人房间 | Go / WebSocket |
| 地图生成 | Go 或 TypeScript |
| 数据传输 | JSON |

## 玩法规则

- 无人机电量耗尽会迫降。
- 浓烟区域传感器失效。
- 幸存者需要先定位，再投递物资。
- 所有幸存者获救或时间耗尽后结算。

## MVP 功能

- 一个 30x30 网格灾区地图。
- 3 架无人机。
- 可点击设置目标点。
- 有电量消耗和视野探索。
- 随机生成 5 个幸存者。
- 救出数量作为得分。

## 数据文件示例

```json
{
  "drones": [
    {"id": "d1", "x": 2, "y": 2, "battery": 100, "cargo": 2},
    {"id": "d2", "x": 3, "y": 2, "battery": 100, "cargo": 1}
  ],
  "tasks": [
    {"drone_id": "d1", "type": "move", "target": [10, 12]}
  ]
}
```

## Codex 开发任务建议

```text
1. 创建 TypeScript + Vite 项目，引入 PixiJS。
2. 绘制网格地图、无人机图标、雾区和幸存者标记。
3. 实现点击无人机和目标点派发任务。
4. 用 Go 实现 WebSocket tick 服务，每 100ms 更新状态。
5. 客户端接收 state_update 并重新渲染。
6. 加入电量、载重、救援成功结算。
```

# 7. 微型高尔夫地形编辑器《Mini Golf Terrain Lab》

## 游戏概念

玩家不是直接打高尔夫，而是先编辑地形：升高坡面、挖坑、放置弹性墙、风扇和传送洞。然后点击测试，看小球是否能一杆进洞。

核心乐趣是“设计物理路径”，像关卡编辑器和解谜游戏的结合。

## 推荐语言组合

```text
Kotlin + libGDX + C++ 物理模块
```

## 可视化前端 / 可操作入口

使用 **libGDX** 做桌面或 Android：

- 触摸或鼠标拖动调整地形高度。
- 工具栏选择坡面、墙、风扇、传送洞。
- 点击小球并拖拽设置初速度。
- 按下 `Test` 后小球开始运动。
- 失败后可以回到编辑模式继续调整。

## 各模块分工

| 模块 | 技术 |
|---|---|
| UI、地图编辑、输入 | Kotlin / libGDX |
| 小球轨迹和碰撞 | C++ 动态库 或 Kotlin 简化物理 |
| 地形保存 | JSON |
| 移动端打包 | Gradle |

## 玩法规则

- 玩家可编辑地形，但每种工具消耗预算。
- 小球只能发射一次。
- 小球进洞则通关。
- 使用更少预算获得更高评分。

## MVP 功能

- 2D 地图编辑网格。
- 可放置墙体和风扇。
- 可拖拽发射小球。
- 小球与墙反弹。
- 小球进入洞口判定胜利。

## 数据文件示例

```json
{
  "level_id": "golf_001",
  "ball_start": [2, 8],
  "hole": [18, 3],
  "objects": [
    {"type": "wall", "x": 8, "y": 6, "rotation": 45},
    {"type": "fan", "x": 10, "y": 5, "force": [1.2, 0]}
  ]
}
```

## Codex 开发任务建议

```text
1. 创建 Kotlin libGDX 项目。
2. 实现编辑模式和测试模式切换。
3. 实现墙体、风扇、洞口、小球实体。
4. 实现鼠标拖拽发射小球。
5. 将地形和物体保存为 level.json。
6. 可选：将物理计算抽成 C++ 动态库。
```

# 8. 摩斯电码海盗电台《Morse Radio Operator》

## 游戏概念

玩家是海上电台员，需要在杂音中接收摩斯电码、判断信号来源、回复正确指令。关卡会模拟雨声、雷暴、干扰台和信号衰减。

这是一款“听觉识别 + 操作反应”的小游戏。它不靠键盘速度，而靠玩家辨认短音、长音和节奏间隔。

## 推荐语言组合

```text
JavaScript / Web Audio API + Rust WebAssembly + Python 内容生成器
```

## 可视化前端 / 可操作入口

使用 **浏览器网页**：

- 中间是老式电台仪表盘。
- 旋钮调整频率和增益。
- 波形区域显示当前音频信号。
- 玩家用按钮输入 `.` 和 `-`，或直接键盘输入。
- 系统显示接收到的片段和解码结果。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 电台 UI、音频播放 | JavaScript / Web Audio API |
| 信号匹配、摩斯解码 | Rust / WASM |
| 关卡文本、干扰参数生成 | Python |
| 前端工程 | TypeScript 可选 |

## 玩法规则

- 每关播放一段摩斯信号。
- 玩家需要调频找到最清晰的信号。
- 输入正确解码文字后通关。
- 干扰越强，得分倍率越高。

## MVP 功能

- 10 个固定摩斯短句。
- 支持播放点和划组成的音频。
- 玩家能输入点划。
- WASM 判断输入是否匹配。
- 显示正确率和用时。

## 数据文件示例

```json
{
  "mission_id": "radio_001",
  "message": "SOS CARGO NORTH",
  "morse": "... --- ... / -.-. .- .-. --. --- / -. --- .-. - ....",
  "noise": 0.35,
  "frequency": 7100
}
```

# 9. 反重力仓库搬运《Gravity Warehouse》

## 游戏概念

玩家在一个仓库里搬运箱子，但可以切换局部重力方向。箱子可能向上、向左或向右坠落。玩家要利用重力变化，把箱子送到指定货架。

它不是机器人流水线，因为玩家是即时操控角色，并且核心机制是“重力方向改变”。

## 推荐语言组合

```text
C# + Unity 2D + YAML 关卡配置
```

## 可视化前端 / 可操作入口

使用 **Unity 2D**：

- WASD 移动角色。
- E 推箱子或抓取箱子。
- 数字键 1/2/3/4 切换房间重力方向。
- 箱子根据当前重力方向滑落或坠落。
- 目标货架高亮显示。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 2D 角色控制、物理 | Unity C# |
| 关卡配置 | YAML / JSON |
| 编辑器扩展 | Unity Editor C# |
| 音效动画 | Unity Timeline 可选 |

## 玩法规则

- 每关有若干箱子和货架。
- 箱子必须按标签送到对应货架。
- 切换重力次数越少，评分越高。
- 有易碎箱子，碰撞过强会损坏。

## MVP 功能

- 一个房间关卡。
- 角色可以移动和推箱子。
- 支持四向重力切换。
- 箱子能落到指定货架。
- 成功后显示评分。

## 数据文件示例

```yaml
level_id: gravity_001
gravity_switch_limit: 12
boxes:
  - id: box_a
    type: fragile
    start: [3, 4]
    target: shelf_a
shelves:
  - id: shelf_a
    position: [12, 2]
```

## Codex 开发任务建议

```text
1. 创建 Unity 2D 项目，实现 PlayerController。
2. 实现 GravityManager，支持 Up/Down/Left/Right 四个方向。
3. 实现 BoxController，根据当前重力方向移动或坠落。
4. 实现 ShelfTrigger，检测正确箱子进入目标区域。
5. 加入切换次数计数和通关评分。
6. 支持从 YAML 或 JSON 加载关卡。
```

# 10. 口袋生态瓶沙盒《Pocket Terrarium》

## 游戏概念

玩家在一个小生态瓶里放入水、土壤、苔藓、小虫、光源和温度调节器。目标不是经营赚钱，而是维持一个微型生态系统稳定。水分太多会发霉，光太少植物枯萎，小虫太多会破坏根系。

这是一个“微型生态模拟 + 沙盒观察”的小游戏，适合做成安静的可视化桌面宠物。

## 推荐语言组合

```text
Flutter / Dart + C++ 模拟核心 + SQLite
```

## 可视化前端 / 可操作入口

使用 **Flutter** 做桌面或移动端：

- 屏幕中心是透明玻璃生态瓶。
- 玩家拖入土壤、水滴、种子、小虫、石头、灯光。
- 底部有温度、湿度、光照滑块。
- 生物和植物会随时间变化。
- 右侧显示生态指标曲线。

## 各模块分工

| 模块 | 技术 |
|---|---|
| 跨平台 UI、拖拽交互 | Flutter / Dart |
| 生态状态模拟 | C++ 动态库 |
| 存档 | SQLite |
| 数据图表 | Flutter Charts 或自绘 Canvas |

## 玩法规则

- 每种生物有适宜温度、湿度、光照范围。
- 环境过偏会导致死亡或爆发式繁殖。
- 玩家需要维持 7 个游戏日生态稳定。
- 不同组合会解锁新的生态瓶主题。

## MVP 功能

- 3 种植物、2 种小虫、3 个环境参数。
- 拖拽添加物种。
- 每秒更新生态状态。
- 显示温度、湿度、氧气、生命数量曲线。
- 支持保存和读取生态瓶。

## 数据文件示例

```json
{
  "terrarium_id": "jar_001",
  "environment": {
    "temperature": 24,
    "humidity": 0.62,
    "light": 0.75
  },
  "species": [
    {"type": "moss", "count": 8, "health": 0.91},
    {"type": "springtail", "count": 12, "health": 0.84}
  ]
}
```

## Codex 开发任务建议

```text
1. 创建 Flutter 项目，设计生态瓶主界面。
2. 实现拖拽添加种子、水滴、小虫等操作。
3. 用 Dart 先实现简化生态更新公式。
4. 后续将 update_ecosystem(state_json) 抽为 C++ 动态库。
5. 使用 SQLite 保存生态瓶状态。
6. 绘制温度、湿度、生命数量随时间变化的曲线。
```
